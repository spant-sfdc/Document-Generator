DocGen – Document Generation Module: Developer Guide

Version: 1.0  |  Last Updated: May 2026  |  Audience: Salesforce Developers

Architecture Overview

DocGen is a layered service architecture. Each layer has a single responsibility and communicates through typed Apex wrapper classes.

Template → Token Extraction → Mapping & Variables → SOQL / Data Resolver → Merge Engine → Rendered Output

Layer Responsibilities

Complete Generation Flow

  1. LWC calls generatePreview(configJson) — serialised TemplateConfigWrapper containing templateId, primaryObject, recordId, tokenMappings[], variables[].
  2. DocGen_DocumentGeneratorService.generateHtml(config):
    1. Loads template HTML from ContentVersion.
    2. Calls DocGen_SmartTokenResolver.mergeWithAutoResolved() to supplement explicit mappings with auto-resolved ones for any unmatched token.
    3. Calls DocGen_DataResolverService.resolveData() to execute SOQL and return primary + child record maps.
    4. Builds a token-alias data map from primary record — adds token names (e.g. total_donation) as aliases for the underlying field values so Boolean variable expressions can reference them.
    5. Calls DocGen_VariableResolverService.resolveWithData() to evaluate all variables.
    6. Calls DocGen_MergeEngineService.mergeTemplate().
  3. DocGen_MergeEngineService.mergeTemplate() in order:
    1. Replaces variable tokens ({{isMajorDonor}}, {{sys.today}}, etc.).
    2. Calls processRepeatingSections() — expands each {{#section}}...{{/section}} block using child record rows.
    3. Calls processConditionals() — evaluates {{#if var}} blocks using merged primary + variable data.
    4. Replaces all remaining primary/parent field tokens.

Key Technical Details

Parent Relationship Resolution

Parent field mappings use a relationship traversal path (e.g. Account.Name) which is added to the primary object's SOQL field list. DocGen_DataResolverService.flattenRecord() uses SObject.getPopulatedFieldsAsMap() — not SObject.get() — to safely traverse relationship keys without runtime exceptions. Nested SObjects are recursively flattened with dot-separated keys (Account.Name).

// Example: SOQL built for Opportunity with Account parent
SELECT Id, Name, Amount, Account.Name, Account.Donor_ID__c
FROM Opportunity
WHERE Id = :recordId
WITH SECURITY_ENFORCED
LIMIT 1

Child Loop Context

Repeating sections are processed by processRepeatingSections(). The method:

  1. Matches the section name against childData (keyed by the token's section name, e.g. line_items).
  2. Calls getSectionMappings() to collect all field mappings for the child SObject. This uses two match strategies:
    • Primary: m.isRepeating == true && m.repeatObject == effectiveObj
    • Fallback: m.sourceObject == effectiveObj — handles mappings where isRepeating was saved incorrectly
  3. For each child record row, resolves each token via row.get(sm.sourceField) — the child record map, never the primary record map.
Critical: Child field mappings must have isRepeating = true and repeatObject = [child SObject API name]. If isRepeating = false, the mapping falls through to the primary-field replacement loop which looks up the field on the primary record (null) and replaces with an empty string. The fallback bySourceObj match in getSectionMappings prevents this for correctly structured mappings.

Token Alias Mapping

Boolean variable expressions reference token names (e.g. total_donation > 10000), not raw field API names (e.g. Amount). DocGen_DocumentGeneratorService.generateHtml() builds a tokenData map that aliases each token name to its mapped field value:

// For mapping: total_donation → Opportunity.Amount
tokenData.put('total_donation', data.primaryRecord.get('Amount')); // e.g. 15000.0

// Variable expression evaluation:
evaluateExpression('total_donation > 10000', tokenData); // → true

Variable Evaluation Flow

  1. DocGen_VariableResolverService.resolve(variables) — handles static types (CONSTANT, TEXT, NUMBER, DATE) and system variables.
  2. resolveWithData(variables, tokenData) — overrides BOOLEAN/EXPRESSION variables by evaluating the expression string against tokenData using evaluateExpression().
  3. Supported operators: >, <, >=, <=, ==, !=, null. Numeric comparison is attempted first; string comparison is the fallback.
  4. Result is a Map<String, String> (variable name → resolved string value).

SOQL Builder & FLS

DocGen_DataResolverService.buildFieldList() validates each requested field against the org schema before adding it to the SELECT list:

  • Dot-separated paths (parent traversal) are validated by validateTraversalPath() which walks relationship names, not field API names.
  • Simple fields are validated via Schema.DescribeFieldResult.isAccessible(). Fields without FLS read access are silently excluded — no exception is thrown, but the field will be absent from SOQL results.
  • All queries use WITH SECURITY_ENFORCED.

Two-Step PDF Generation

PDF generation is split into two separate @AuraEnabled transactions to avoid a Salesforce platform constraint: PageReference.getContentAsPDF() fires an internal HTTP request which cannot see uncommitted DML in the same transaction.

  1. Step 1 — storeMergedHtml(configJson): Generates the merged HTML, inserts it as a ContentVersion, and commits. Returns the ContentVersion Id.
  2. Step 2 — generateDocument(configJson, 'PDF', linkedRecordId, tempCvId): Reads the committed ContentVersion, passes it to the DocGen_PdfRenderer Visualforce page via URL parameter, captures the PDF, saves it as a file linked to the record, then deletes the temp ContentVersion.
